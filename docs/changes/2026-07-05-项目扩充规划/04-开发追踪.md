# 开发追踪：A 主线逐步完成并校验

## 开发顺序

1. M0 调研与约束：冻结范围、引用来源、并行规则。
2. M1 模板系统：模板注册表与变量填充。
3. M2 基础后端：日期范围聚合 + report API + 周报/月报生成。
4. M4 Word 渲染与校验：报告 docx、变量/字数/空内容校验。
5. M2 Web 入口：Console 选择报告类型、日期范围、模板、下载。
6. M2 Android 入口：App 端报告生成/下载。
7. M3 实习总结：长范围聚合、总结模板、导出。
8. 集成回归：后端 pytest、Android assembleDebug、手工 API smoke。

## 并行波次

### Wave 0：只读 scout（可 4 个并行）
- Backend Scout：只读 `intern-diary-server/app/*.py` 和 tests，返回 API/存储影响面。
- Android Scout：只读 `intern-diary-android/app/src/main`，返回最小 UI 接入点。
- Renderer Scout：只读 `word_renderer.py`、`templates/`，返回模板变量改造点。
- QA Scout：只读 tests/docs/build 文件，返回最小验收命令和缺口。

### Wave 1：基础实现（最多 3 个并行，写入集隔离）
- Backend Worker：写 `schemas.py`、`paths.py`、`model_client.py`、`main.py` 中 report API 和生成逻辑。
- Renderer Worker：写 `word_renderer.py`、`templates/report_templates.json`、必要模板样例。
- Test Worker：先写后端单测草案；不得改生产代码。

### Wave 2：UI 实现（后端 API 契约稳定后，2 个并行）
- Web Worker：写 `/console` 内报告 UI 和 JS。
- Android Worker：写 `ApiClient.kt`、`TodayActivity.kt`、相关 layout/string。

### Wave 3：实习总结 + 回归（2-3 个并行）
- Summary Worker：基于周/月报能力扩展总结生成。
- QA Worker：跑 pytest、Android build、curl smoke，整理失败。
- Docs Worker：更新 README/usage/api 文档；不得改代码。

## 循环依赖告警

- Android/Web 入口依赖 Report API；不要提前写死未定字段。
- Renderer 与 Backend 共享 report metadata；先在 `07-API文档.md` 固化字段。
- 测试可并行写，但最终以生产 API 字段为准统一修正。
