# 需求文档：A 主线优先的全量扩充路线

> 用户选择 D（全都要），但第一阶段选 A：实习成果生成器。本文把 D 收进长期路线图，把 A 拆成可逐步完成和校验的第一主线。

## 产品定位

`diary-open-source` 从“Android + FastAPI 实习日记生成器”扩充为“实习成果生成器”：用户每天采集素材，系统可生成日报、周报、月报、实习总结，并按模板导出 Word。

## 用户场景

| 场景 | 用户 | 想达成 | 触发时机 |
|---|---|---|---|
| S1 日报 | 实习生 | 从当天素材生成正式日记 Word | 每天下班/睡前 |
| S2 周报 | 实习生 | 自动汇总一周工作、问题、收获、下周计划 | 每周五/周末 |
| S3 月报 | 实习生/学生 | 汇总月度任务、技术成长、成果证据 | 月末 |
| S4 实习总结 | 实习生/学校提交 | 按学校/公司模板生成完整总结 | 实习结束前 |
| S5 模板切换 | 不同学校/岗位用户 | 使用自己的 Word 格式、字段和字数要求 | 首次部署或提交前 |
| S6 校验重生成 | 实习生 | 发现漏项/格式不符后快速修正 | 导出前 |

## 分阶段路线

### Phase A0：现状固化与回归基线
- 明确当前日报生成、图片描述、Word 渲染、Android 提交流程的测试基线。
- 不新增用户功能，只补最小回归用例和文档入口。

### Phase A1：模板系统
- 支持模板清单、模板变量、报告类型默认模板。
- 先文件配置，不上数据库。

### Phase A2：周报/月报
- 新增报告类型 `weekly` / `monthly`。
- 输入日期范围，输出 Markdown + Word。

### Phase A3：实习总结
- 新增报告类型 `internship_summary`。
- 支持更长日期范围与更长输出。

### Phase A4：导出校验 + UI 完整化
- Web + Android 两端走通。
- 校验未填变量、空输出、字数过短、LLM 失败重试提示。

### Phase D-Later：全量扩展池
- 标签/搜索/统计、语音/离线/提醒、多用户/PWA/备份、长期 AI 问答。
- 只在 A 主线稳定后进入。

## PM 实用性判断

- A 主线最贴合“实习日记助手”的差异化：多数日记 App 能记，少数能稳定生成学校/公司可提交成果。
- 第一阶段不做大而全，避免项目变成泛用 journal clone。
- 先复用现有文件存储和 Word 渲染，避免数据库迁移拖慢价值交付。
